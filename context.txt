High-level project overview
This is a Python 3.11 project named tailscale-dml. It is a Tailscale-connected
distributed machine learning demo with a leader process and worker processes.
Workers connect to the leader over an asyncio JSON-lines control protocol, send
heartbeats, report hardware, receive training tasks/shards, and run ML training.

The project supports two training modes:

1. federated
   - MNIST.
   - Leader assigns contiguous mini-batch ranges to workers.
   - Workers train a small MLP locally.
   - Workers return model weights.
   - Leader aggregates successful worker models with weighted FedAvg by sample
     count.
   - This mode naturally tolerates workers joining/leaving between rounds.

2. distributed
   - CIFAR-10.
   - Leader participates as rank 0.
   - Workers are additional ranks.
   - Leader splits data shards by benchmark score and sends worker shards as
     binary payloads.
   - Participants use torch.distributed with the gloo backend.
   - Training uses all-reduce of flattened gradients.
   - Membership is effectively frozen per distributed run because
     torch.distributed world_size and ranks are fixed. New workers that connect
     mid-run should wait for the next run.

Protocol notes
The control channel uses newline-delimited JSON objects. Binary payloads are
used for distributed CIFAR shards.

Common message types:
- Worker to leader: hello, heartbeat, worker_log, train_result, train_error,
  distributed_shard_ready, distributed_epoch, distributed_error,
  distributed_complete.
- Leader to worker: welcome, reject, shutdown, train_task, distributed_shard,
  distributed_train_start.

Protocol limits:
- JSON stream limit: 64 * 1024 * 1024.
- Binary payload limit: 512 * 1024 * 1024.

Federated mode details
Leader behavior:
- Requires at least one connected worker.
- Creates an initial MNIST model payload with seed 1337.
- At each epoch boundary, snapshots currently connected workers.
- Allocates contiguous MNIST batch ranges proportional to worker
  benchmark_score.
- Sends each worker a train_task containing model state and batch assignment.
- Collects train_result messages until timeout.
- Reassigns lost work to currently connected workers when possible.
- Uses weighted FedAvg by sample count.
- Evaluates on MNIST test data.

Worker behavior:
- Downloads MNIST under .data.
- Selects cuda, then mps, then cpu.
- Trains MnistNet, a simple flatten plus two linear layers.

Distributed mode details before latest local fix
Original behavior:
- dml_cluster.distributed_training.run_training called run_one_epoch once per
  epoch.
- Each epoch initialized a new torch.distributed process group.
- Each epoch destroyed the process group.
- This worked for one epoch but failed on epoch 2 in local loopback tests with
  worker error "dist init: failed to recv, got 0 bytes" and leader timeout
  "1/2 clients joined".

Distributed mode latest local fix
Changed file: dml_cluster/distributed_training.py

New behavior:
- run_training initializes torch.distributed once per distributed run.
- CifarCnn is constructed once.
- Adam optimizer is constructed once.
- DataLoader is constructed once.
- Rank 0 broadcasts the initial model once.
- All epochs run inside the same process group.
- Gradients are still all-reduced per batch.
- Loss is still all-reduced per epoch.
- Barriers are used after startup, after each epoch, and before final complete.
- Process group is destroyed once at the end.
- Result queue emits distributed_epoch for every epoch and
  distributed_complete at the end.

Important distributed-mode policy
- New workers can connect to the leader while a distributed run is active, but
  they cannot join that active torch.distributed group because rank assignment
  and world_size are fixed.
- For stable distributed mode, snapshot workers at run start, freeze membership
  for that run, and let late workers wait for the next run.
- If a participating worker drops mid-run, the distributed all-reduce run should
  fail/stop cleanly. FedAvg mode is the better mode for dynamic worker churn.

- The other repo and our distributed mode both use CIFAR-10 + gloo +
  all-reduce gradient synchronization.

- FedAvg is the "other method" compared with all-reduce: workers train local
  model copies, send weights back, and the leader averages model weights by

Worker project_dir bug fix (2026-05-07)
File: dml_cluster/distributed/data_parallel/worker_session.py

Bug: handle_distributed_shard_config stored the incoming shard config dict
verbatim, including project_dir which was set to the LEADER's filesystem path
(e.g. /Users/ruturaj_vasant/Desktop/Academic/DS_Project). When the worker's

Symptom in logs: "distributed error: dataset load: [Errno 13] Permission denied:
'/Users/ruturaj_vasant'"

Fix: immediately after storing the dict, override project_dir with
str(worker.project_dir) so each participant loads datasets from its own
runtime directory. Commit 48b95b3.

Gloo interface fallback fix (2026-05-07) — commit 9975fbc
Files: dml_cluster/distributed/common/network.py
       dml_cluster/distributed/data_parallel/runner.py
       dml_cluster/distributed/pipeline_parallel/runner.py

Bug: Two related bugs caused the worker to enter dist.init_process_group on
the wrong network interface (e.g. NYU wireless 10.19.x.x instead of Tailscale),
making rank 0's Gloo peer-connection to rank 1 unreachable. Rank 0 was stuck in
CONNECTING state and the Gloo cleanup assertion fired:
  gloo::EnforceNotMet: Cannot close pair while waiting on connection (SIGABRT)

Bug 1 (network.py create_gloo_pg_options): When ProcessGroupGloo._Options or
create_device() raised an exception, or when those attrs were missing entirely,
the function returned (None, tailscale_ip) WITHOUT setting GLOO_SOCKET_IFNAME.
Fix: added _set_gloo_socket_ifname_fallback(tailscale_ip) which is called from
all three early-exit paths (ProcessGroupGloo absent, create_device/_Options
absent, exception in options setup).

Bug 2 (runner.py, both data_parallel and pipeline_parallel): The branch condition
was `if gloo_hostname` (truthy whenever Tailscale is found), so configure_
gloo_socket_ifname was NEVER called when Tailscale IP existed but pg_options
failed. Changed to `if pg_options is not None` so the GLOO_SOCKET_IFNAME
fallback is always applied when explicit options are absent.

After these fixes:
- If pg_options creation succeeds: rank uses explicit Gloo device (most reliable)
- If it fails: GLOO_SOCKET_IFNAME is set to the Tailscale interface (fallback)
- Either way, Gloo does NOT pick an unrelated interface like NYU wireless

Reliability and execution improvements (2026-05-07)
These changes were diagnosed and applied after a review of the distributed code
path, targeting failures that would occur on real multi-machine runs with
ResNet101 and TinyImageNet.

Problem 1: Dataset download inside training thread caused Gloo broadcast timeout.
Before this fix, run_training() called _build_local_dataset_loader() AFTER
dist.init_process_group(). If the worker needed to download TinyImageNet
(237 MB), rank 0 (leader) would already be blocking on dist.broadcast() for the
initial model weights while rank 1 (worker) was still downloading the dataset.
The 60-second Gloo collective timeout would fire before the worker reached
dist.broadcast().
Fix: dml_cluster/distributed/data_parallel/runner.py now builds the DataLoader
(calling load_dataset, including any download) BEFORE calling
dist.init_process_group(). A failed dataset load now emits distributed_error
immediately and returns without ever entering the process group. Dataset load
errors are logged as "dataset load: <exc>" to distinguish them from rendezvous
errors.

Problem 2: Default Gloo collective timeout was 60 seconds — too short for
ResNet101 (176 MB model broadcast) on slow links and for multi-machine runs.
Fix: DEFAULT_DIST_TIMEOUT_SECONDS in runner.py raised from 60.0 to 120.0.
--distributed-timeout default in leader.py CLI raised from 60.0 to 120.0.
experiments/run_experiments.sh now passes --distributed-timeout with a default
of DIST_TIMEOUT=300 for real runs, and 120 for --loopback mode.

Problem 3: wait_for_worker_distributed_completion was capped at 10 seconds.
After the leader's training thread finished, it only waited min(10, timeout)
seconds for workers to send distributed_complete. For long runs or slow links
this caused "distributed run ended before completion messages" warnings and
lost the final worker metrics row in the CSV.
Fix: dml_cluster/distributed/data_parallel/leader_session.py now waits
min(60.0, max(10.0, timeout_seconds * 0.5)) seconds, scaling with the run
timeout up to a 60-second cap.

Problem 4: TopK gradient compression issued 2×N_params Gloo collectives per
batch. For ResNet101 with ~339 named parameters this was 678 separate
all_gather calls per batch, each with Tailscale RTT overhead.
Fix: _topk_allreduce_grads() in runner.py now collects all per-parameter
compressed payloads first, concatenates them into two flat tensors (indices and
values), does exactly 2 dist.all_gather calls for the entire batch, then scatters
the reduced values back to each parameter using offset slices.

Problem 5: Same dist port reused across all 5 experiments in run_experiments.sh.
TCP TIME_WAIT state after dist.destroy_process_group() could block the next
experiment from binding the port.
Fix: run_experiments.sh now passes a unique --dist-port per experiment:
EXP 1: DIST_PORT+0, EXP 2: DIST_PORT+2, ..., EXP 6: DIST_PORT+10.
The base DIST_PORT is still passed in leader_base but overridden per experiment
via argparse last-wins semantics.

- New CLI option:
  --distributed-parallel parameter-server
- Parameter-server mode avoids torch.distributed/Gloo. It uses the existing
  leader-worker TCP control channel on port 8787 plus binary payloads:
  leader sends parameter_server_round_start + model state_dict bytes.
  worker trains its local shard for the configured batches/epoch.
  worker returns parameter_server_update + updated state_dict bytes.
  leader aggregates states weighted by sample count and evaluates rank 0.

  --distributed-optimizations none
  --distributed-optimizations fp16
  fp16 serializes floating state_dict tensors as float16 for model download and
  worker upload, typically reducing communication by about 2x.

  parameter-server CSV fields include model_download_bytes, update_upload_bytes,
  total_communication_bytes, model_download_mb, update_upload_mb,
